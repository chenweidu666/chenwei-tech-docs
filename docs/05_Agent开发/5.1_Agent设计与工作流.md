# 1. 什么是 Agent


- **定义（工程视角）**：在 **循环（Observe → Think → Act）** 中驱动大模型，使其能够 **读取环境信息、调用外部工具、更新状态**，直至满足终止条件（任务完成、步数上限、人工介入等）。  
- **与「单次问答」的区别**：问答多为 **单轮或浅多轮**；Agent 强调 **多步决策**、**副作用**（写库、下单、发消息）与 **可恢复的执行轨迹**。  
- **适用边界**：适合目标可分解、工具可靠、反馈可观测的任务；对 **强实时硬约束、强合规审计、不可回滚操作** 需额外的人审与幂等设计。  
- **常见宿主形态**：聊天应用内的「智能体」、定时触发的 **自主任务（7×24）**、IDE/运维场景下的 **Coding Agent** 等。

# 2. 主流 Agent 模式


## 2.1. ReAct（推理 + 行动）

- **思路**：显式交替输出 **思考（Thought）** 与 **行动（Action）**，行动结果为 **观察（Observation）**，再进入下一轮，直到给出 **最终答案（Answer）**。  
- **优点**：轨迹可解释、易调试；与 **Tool Calling** 天然契合。  
- **注意**：要限制 **最大步数**、防止「空转调用」；对工具返回要做 **长度截断与结构化摘要**，避免上下文爆炸。

## 2.2. Plan-and-Execute

- **思路**：先 **规划** 出阶段目标或子任务列表，再 **按阶段执行**；执行中可 **重规划（Re-plan）**。  
- **优点**：适合长链路任务（研究报告、流水线式生产）；比纯 ReAct 更易做 **并行子任务**。  
- **注意**：规划幻觉、步骤依赖遗漏；需 **依赖检查** 与 **失败回滚策略**。

## 2.3. 多 Agent 协作

- **典型切分**：按 **角色**（规划 / 检索 / 工具 / 校验）、按 **领域**（数据 / 策略 / 通知）或按 **流水线阶段** 拆成多个 Agent。  
- **编排方式**：中心化 **协调者（Orchestrator）**、**层级汇报**、或 **图状态机（如 LangGraph）** 显式建模状态与边。  
- **工程要点**：**单一事实来源（SSOT）** 的状态存储、**消息契约（schema）**、避免多 Agent 重复写同一资源；评估上要定义 **跨 Agent 的指标**（成功率、成本、时延）。  

与 **MCP** 的关系：多工具、多数据源时，用标准协议收口工具发现与调用，见 [5.2 MCP 协议](5.2_MCP协议.md)。

# 3. 工具调用设计


## 3.1. 工具定义

- 使用 **JSON Schema**（或宿主平台等价物）描述 **名称、参数、返回值含义**；枚举、默认值、必填项写清楚，减少模型胡填参数。  
- **幂等与安全**：读操作与写操作分开；写操作尽量 **带 request_id / 业务幂等键**；敏感操作预留 **Human-in-the-Loop**。

## 3.2. 工具路由

- **粗分层**：先 **意图/领域分类** 再进工具集，避免一次性塞给模型上百个工具描述。  
- **动态子集**：按会话状态只挂载当前阶段可用工具（例如「仅行情」「仅账户」）。  
- **超时与重试**：为每类工具设置 **超时、重试次数、熔断条件**；区分可重试错误（网络）与不可重试（业务拒绝）。

## 3.3. 结果处理

- **结构化优先**：表格/JSON 片段优于长篇自然语言堆给下一轮。  
- **校验**：类型、范围、空值；不通过则 **明确错误码** 反馈给模型重试或改参。  
- **观测性**：记录 **tool_name、latency、status** 与摘要结果，便于排障与评估。

# 4. Agent 记忆体系


## 4.1. 短期记忆（对话上下文）

- **窗口管理**：滑动窗口、摘要压缩（递归摘要）、或 **关键事实提取** 写入结构化槽位。  
- **系统提示稳定性**：将「不变约束」放系统侧，将「易变状态」放记忆层，避免每次全量重复。

## 4.2. 长期记忆（持久化存储）

- **向量库 / 文档库**：适合事实、偏好、历史案例检索；注意 **权限与脱敏**。  
- **结构化存储**：用户画像、策略参数、风控规则等用数据库或 KV，便于事务与审计。

## 4.3. 跨会话状态

- **会话外状态机**：任务 ID、当前阶段、未完成的子任务列表；进程重启后可 **hydrate**。  
- **日志与 WAL 思路**：按日或按 run 追加 **决策日志 / 工具调用日志**，支持复盘与重放；与「仅聊天历史」解耦，避免把日志全塞进模型上下文。  
- **一致性**：多进程或分布式时，用 **乐观锁 / 版本号** 避免并发写乱序。

# 5. 异常处理与熔断


- **层级**：工具层错误 → 模型重试/换参；Agent 层步数耗尽 → 降级为「只读建议」或转人工。  
- **熔断**：连续失败率超阈暂时 **禁用某工具或某数据源**，并写入告警渠道。  
- **降级路径**：主模型不可用时的 **备用模型**、关闭非关键工具、只保留查询类能力。  
- **可观测性**：统一 **trace_id**；关键路径打 **结构化日志**；对外部 API 做 **配额与限流**。

# 6. 实战：GateClaw — 全自主 BTC 现货交易 Agent


GateClaw 是我 2026 年 3 月搭建的个人量化交易 Agent，运行在 OpenClaw 框架上，7×24 无人值守执行 BTC 现货金字塔策略。以下是真实架构的工程拆解。

## 6.1. 整体架构

```
定时触发（Cron 每小时整点）
       ↓
  采集层（4个MCP Server）
  ├── gate-market  → 实时价格、资金费率、多空比、清算数据
  ├── gate-info    → RSI / MACD / 综合技术信号
  ├── gate-news    → 社交情绪评分、最新事件
  └── gate         → 账户余额、挂单状态、成交记录
       ↓
  决策层（LLM + 策略规则）
  ├── 分析：信号是否触发买入/撤单条件
  ├── 规则：BTC>$73,000 → 市价追买50U
  │        BTC<$60,000 → 撤全部挂单+告警
  │        RSI(4h)>70  → 卖出目标+7%
  │        成交        → 立即挂卖单×1.05
  └── 判断：偶数小时 → 发完整汇报；有异常 → 立即通知
       ↓
  执行层（gate MCP Server）
  ├── cex_spot_create_order   → 挂买单/卖单
  ├── cex_spot_cancel_order   → 撤单
  └── Telegram 通知
       ↓
  记忆层（文件系统）
  ├── MEMORY.md        → 长期记忆（策略参数、里程碑、关键决策）
  ├── memory/YYYY-MM-DD.md → 每日日志（决策记录、成交记录）
  └── HEARTBEAT.md     → 心跳任务定义（相当于Agent的SOP）
```

## 6.2. 关键工程决策

**MCP Server 按域拆分**
- 4个独立Server：`gate`（账户操作）/ `gate-market`（行情）/ `gate-info`（技术分析）/ `gate-news`（情绪）
- 好处：权限隔离（只有`gate`能下单）、便于单独调试

**策略与LLM解耦**
- 买卖条件写成明确规则（非LLM推断），避免模型"发挥"导致误操作
- LLM负责：信息整合、异常识别、汇报生成
- 规则引擎负责：触发判断与执行动作

**记忆分层设计**
- `MEMORY.md`：长期约束，如"100U保证金永远不动"，每次启动都加载
- 每日日志：追加写，不覆盖，支持事后审计
- `HEARTBEAT.md`：定义SOP，相当于Agent的"操作手册"，更新后立即生效

**可靠性设计**
- 每档挂单有唯一订单ID，定期校验是否仍在open状态
- 通知优先级：成交 > 触发条件 > 偶数小时汇报 > 静默
- 异常直接Telegram报警，不静默吞掉

## 6.3. 踩过的坑

- **MCP参数格式**：mcporter调用gate-info时，`{"symbol":"BTC"}` JSON格式会报"symbol不能为空"，必须用 `symbol=BTC` 的key=value格式
- **历史commit含手机号**：简历站重建时发现git历史有61个commit含敏感信息，最终选择删库重建（1个干净commit）
- **Cron配置方式**：手动编辑`jobs.json`不会被Gateway动态加载，必须用`openclaw cron add`CLI注册
