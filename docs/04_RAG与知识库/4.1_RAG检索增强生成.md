<div style="text-align: center; font-size: 2rem; font-weight: 700; margin-bottom: 0.5rem;"><strong>RAG、Milvus 与 LangChain 知识工程</strong></div>

本篇将 **检索增强生成（RAG）**、**Milvus 向量库** 与 **LangChain 编排** 放在同一叙事下：先建立端到端流程，再展开存储与框架落地，便于对照阅读与实施。

# 1. 为什么需要 RAG

大模型的预训练知识 **静态、有截止日期**，且易产生 **幻觉**。RAG 在生成前从 **外部知识库** 检索与问题相关的片段，拼入提示词，使回答 **可溯源、可更新**，适合企业文档、客服、内部规章等场景。代价是增加 **检索质量、延迟与运维** 复杂度，需与纯微调、长上下文等方案按成本与合规综合选型。

# 2. RAG 完整流程

## 2.1. 文档加载与解析

从 PDF、HTML、Markdown、扫描件等加载正文；扫描件需 **OCR**（如 PaddleOCR、Tesseract）。清洗阶段去除噪声页眉页脚、乱码与重复模板，并保留 **标题层级、章节、来源 URL** 等元数据，供后续切块与引用展示。

## 2.2. 文档切块策略

按 **固定 token/字符窗口**、**语义边界** 或 **结构化标题** 切分；**overlap** 可减少边界截断导致的语义断裂。切块过小丢失上下文，过大则检索噪声多；需在 **评测集** 上网格搜索 chunk 大小与 overlap。

## 2.3. 向量化（Embedding）

用与线上一致的 **嵌入模型** 将 chunk 与 query 映射到同一向量空间；中英文场景可选 **bge**、**m3e**、厂商 Embedding API 等。注意 **维度、归一化方式** 与向量库 **度量（cosine / IP / L2）** 一致。

## 2.4. 向量存储与索引

向量与元数据写入向量数据库（见 **第 4 节 Milvus**），构建 **HNSW、IVF** 等 ANN 索引；写入与检索的 **一致性级别** 按业务延迟要求选择。

## 2.5. 检索策略

**Top-K 相似检索** 为基础；可叠加 **分数阈值、MMR 去冗、元数据过滤**（租户、文档类型）。复杂场景使用 **混合检索**（见第 3 节）与 **Rerank** 提升Precision。

## 2.6. 生成（LLM）

将检索片段与用户问题按模板组装为提示，约束 **「仅依据上下文作答、不知则拒答」**；可要求输出 **引用编号** 便于审计。控制 **max_tokens、temperature**，并对长上下文做 **截断优先级**（如按 rerank 分数排序后装入窗口）。

# 3. 混合检索与重排序

## 3.1. 向量检索与关键词（BM25）

- **向量检索**：捕捉语义近邻，同义词、改写友好。  
- **BM25**：关键词与专有名词匹配强，适合表格、型号、条款编号。  
- **融合**：RRF（倒数排名融合）或加权合并；权重依赖业务语料，需离线对比。

## 3.2. 重排序（Rerank）

先粗召回较大 **K**，再用 **Cross-Encoder**（如 bge-reranker）对 (query, passage) 打分，取 Top-N 送入 LLM，可显著降低 **噪声进窗**。注意延迟与批次大小，必要时异步或缓存热门 query。

## 3.3. 查询侧增强（可选）

**Multi-Query**（多查询改写合并结果）、**HyDE**（先生成假设答案再检索）等可缓解表述漂移；实现成本与收益需在验证集上衡量，避免过度复杂。

# 4. Milvus 向量数据库

## 4.1. 作用与定位

在 RAG 链路中，文档经嵌入得到 **高维向量**，检索即 **ANN（近似最近邻）** 查询。Milvus 负责 **持久化、索引、过滤、扩缩容与运维**，支撑亿级规模与线上 SLA。相对关系型 B-tree，向量库为 **cosine / L2 / IP** 等度量优化；相对内存 Faiss，Milvus 提供更完整的 **服务化与分布式** 能力。

## 4.2. 核心概念

- **Collection**：逻辑「向量表」，含 **Field**（主键、向量、标量元数据等）。  
- **Partition**（可选）：冷热分层或权限隔离。  
- **Index**：ANN 结构；在 **召回、构建时间、内存** 间权衡。

### Collection / Field / Index 设计要点

- **Field**：`FLOAT_VECTOR` 与 `VARCHAR`、`INT64` 等过滤字段（如 `tenant_id`）一并设计。  
- **Schema**：主键稳定可回溯；过滤条件尽量下推到库内 **expr**，避免全表扫。  
- **一致性**：`consistency level` 与读写延迟期望对齐。

### 索引类型对比

| 索引 | 原理 | 适用场景 |
|------|------|---------|
| HNSW | 分层小世界图 | 高精度、延迟敏感、内存充足 |
| IVF_FLAT | 倒排聚类 + 精确重算 | 大规模，调 `nlist` / `nprobe` |
| IVF_SQ8 | 聚类 + 标量量化 | 省内存、可接受轻微精度损失 |
| IVF_PQ | 积量化 | 超大规模、强省内存 |

**实践**：度量与嵌入训练一致；用真实查询集扫 **nprobe、ef** 等参数。

## 4.3. Python SDK 流程（Milvus 2.x）

以下步骤以 **`pymilvus`** 为准，细节以官方文档当前版本为准。

- **连接与建表**：`connections.connect`，定义 `CollectionSchema` / `FieldSchema`，创建后 **`load()`** 再查询。  
- **插入**：批量控制大小；维护 **业务主键 ↔ 内部 id** 映射。  
- **建索引**：`create_index` 指定类型与参数（如 HNSW 的 `M`、`efConstruction`）。  
- **检索**：`search` 传入 query 向量、`top_k`、**expr** 过滤；根据 **distance 含义** 解释排序。

## 4.4. 在 RAG 中的链路位置

1. 文档切块 → 元数据（标题、时间、来源）。  
2. 嵌入 → 与线上一致模型与归一化。  
3. 写入 Milvus → 带租户等过滤字段。  
4. 检索 → top_k + 阈值 + 可选 **reranker**。  
5. 拼 prompt → 调用 LLM。

与 **第 5 节 LangChain** 对接时，可用向量 store 适配器；**schema 与多租户策略** 仍须在业务层设计清楚。

## 4.5. 性能调优

- 网格搜索 **nprobe（IVF）、ef（HNSW）** 与 **嵌入维度**。  
- 读多写少时 **副本** 与分段；写入与全量建索引 **错峰**。  
- 热门 query 向量可 **缓存**，注意租户隔离与失效。

## 4.6. Milvus 踩坑摘要

- 度量与嵌入不一致导致排序错误。  
- 未建索引或未 **load** 导致极慢或失败。  
- **expr** 与字段类型不匹配。  
- 主键重复 **静默覆盖**；Milvus 1.x → 2.x **API 不兼容**，迁移读官方指南。

# 5. LangChain 工程实践

## 5.1. 定位

**LangChain**（Python / JS）提供 LLM 应用的 **组件与编排**：模型 I/O、提示模板、文档加载、向量库、链路与 Agent。不负责训练模型，侧重 **RAG / Agent 工程化**；版本迭代快，生产应 **锁定依赖** 并抽象领域层，避免与框架深度耦合。

## 5.2. 核心模块

- **Model I/O**：`ChatModel` / `LLM`，`PromptTemplate`，输出解析器（JSON / Pydantic）。  
- **Data Connection**：Loaders、Splitters、**Vector stores**（对接 Milvus、Chroma、FAISS 等）。  
- **Chains / LCEL**：`检索 → 拼 prompt → 模型 → 后处理` 的固定流水线。  
- **Agents & Tools**：ReAct、OpenAI tools 等；须 **限步数、超时与权限**（与 [5.1 Agent 设计与工作流](../05_Agent开发/5.1_Agent设计与工作流.md) 呼应）。  
- **Memory**：多轮缓冲或摘要；长会话需 **外部会话存储**。

## 5.3. RAG 链示例（概念）

```python
from langchain.chains import RetrievalQA
chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever()
)
```

具体 import 与 API 以当前 LangChain 大版本文档为准。落地要点：`retriever` 的 **k、MMR**；自定义 **prompt** 约束依据上下文作答；生产侧 **引用 id、拒答策略** 与租户过滤。

## 5.4. Agent 与 MCP

ReAct 类 Agent 需明确 **系统提示、最大迭代次数**。可将 **MCP 工具** 暴露为 LangChain Tool，参见 [5.2 MCP 协议](../05_Agent开发/5.2_MCP协议.md)。

## 5.5. LangChain 踩坑摘要

- 0.x → 1.x **API 大变**，勿盲抄旧教程。  
- 异步路由用 **`ainvoke` / `astream`**，避免阻塞事件循环。  
- **Retriever 未带租户条件** 可致跨租户泄漏。  
- 链路过长难测：拆 **子链** 并对检索与模型 **mock**。

# 6. 规模化落地要点

- **数据**：去重、版本化、更新策略（全量 / 增量）与回滚。  
- **评测**：固定 **Q/A 集 + 引用命中率 + 人工抽检**；上线后持续采样 bad case。  
- **安全**：知识库 **权限模型**、日志 **脱敏**、对外答案 **合规审核**（尤其金融、医疗场景）。  
- **成本**：嵌入批处理、缓存、rerank 批次与 **token 预算** 联动监控。

# 7. 踩坑汇总（全流程）

- **切块与表格**：表格、列表被切碎导致检索丢条件；可用结构化解析或更大块 + 后处理。  
- **查询与文档嵌入模型不一致**：检索效果断崖下降。  
- **上下文塞满噪声**：只加 top_k 不加 rerank 或阈值。  
- **无拒答**：上下文不足时模型胡编；须在提示词与后处理双重约束。  
- **Dify 等产品**：见 [5.3 Dify 工作流平台](../05_Agent开发/5.3_Dify工作流平台.md) 与本文第 2～3 节对照。
