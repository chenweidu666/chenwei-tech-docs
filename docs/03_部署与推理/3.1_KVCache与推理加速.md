# 1. KV Cache 原理与优化

## 1.1. 是什么

自回归生成第 \(t+1\) 个 token 时，前面各层的 **K、V** 对已生成前缀不变，可缓存下来，避免每步重复计算整段历史——这就是 **KV Cache**。没有 Cache 时，每步注意力都要对全长序列重算，延迟随长度平方级恶化，线上几乎不可用。

## 1.2. 显存占用直觉

大致与 **层数 × 序列长度 × 每 token 的 K/V 表示大小 × 精度** 成正比；标准 MHA 还与头数相关，**GQA / MLA** 等会改变「每 token K/V」的有效宽度。上下文从 4K 拉到 128K 乃至 1M 时，KV 往往成为首要瓶颈，并与 batch、并发请求数共同决定 **显存峰值**。

## 1.3. 常见优化方向

- **GQA / MQA / MLA**：从缩头、分组到潜在压缩，降低 Cache 体积（MLA 见 [Transformer 架构详解](../01_基础层/1.2_Transformer架构详解.md) §1.4）。
- **PagedAttention（vLLM）**：把 KV 分页管理，减少碎片与浪费（详见 [3.2 vLLM 推理部署](3.2_vLLM推理部署.md)）。
- **量化 KV**：INT8/FP8 等降低单元素占用（需权衡精度与误差累积）。

# 2. 推理加速关键技术

## 2.1. 批处理与调度

**Continuous Batching**（动态批处理）在请求到达与完成时动态拼 batch，提高 GPU 利用率，降低尾延迟；vLLM、部分商业推理框架均围绕此类调度做优化。

## 2.2. 算子与图优化

FlashAttention 系列降低注意力显存与内存带宽压力；**算子融合**减少 kernel launch 次数。部署时需关注 **CUDA / PyTorch / 推理引擎** 版本组合是否启用对应内核。

## 2.3. 投机解码（Speculative Decoding）

用小模型（草稿）快速生成候选 token，再由大模型**并行验证**，在可接受质量下提高吞吐；实现复杂，需框架与模型支持，且草稿与目标模型分布需匹配。

## 2.4. 权重量化

W8A16、GPTQ、AWQ 等降低权重带宽与显存占用，与 [3.3 INT 量化](3.3_INT量化.md) 衔接。注意量化后 **KV 与激活** 是否一并优化，以及对长上下文的误差影响。
