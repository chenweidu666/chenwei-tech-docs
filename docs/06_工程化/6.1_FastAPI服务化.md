# 1. 为什么用 FastAPI

**FastAPI** 基于 Starlette 与 Pydantic，提供**异步优先**的 HTTP 接口与自动 OpenAPI 文档，非常适合封装 **LLM 推理、RAG 检索、Agent 编排** 等微服务。

- **类型提示 + 校验**：请求体与响应用 Pydantic 模型，减少手写校验代码。
- **异步 `async def`**：可在单 worker 内并发等待 IO（模型 HTTP、向量库、数据库），提高吞吐。
- **生态**：与 Uvicorn、Gunicorn、Docker、K8s 集成成熟。

# 2. 封装 LLM 推理服务

## 2.1. 同步接口

短请求或内部批处理可用同步路由：在路由内直接调用阻塞式 SDK。注意**线程池**隔离，避免阻塞事件循环；GPU 推理若占满单进程，需用 **队列 + worker 进程** 或独立推理引擎（vLLM、TGI）。

## 2.2. 流式响应（SSE）

对话产品常用 **Server-Sent Events**：边生成边推送 token，降低首字延迟。FastAPI 可用 `StreamingResponse`，yield 文本块并设 `media_type="text/event-stream"`。需处理**客户端断开**与**上游流异常**，避免僵尸生成。

## 2.3. 并发控制

- **限流**：按 API Key、IP 或租户限流，保护 GPU 与下游。
- **超时**：对模型与检索设置 `timeout`，返回可诊断错误码。
- **背压**：队列满时拒绝或降级（缓存回答、简短回复），避免 OOM。

# 3. 生产部署

- **Uvicorn + Gunicorn**：多 worker 跑 CPU 绑定逻辑；**GPU 推理**通常 **每卡一进程** 或统一经 sidecar 推理服务，API 层只做编排。
- **Nginx 反向代理**：TLS 终结、长连接与 SSE 缓冲配置、请求体大小限制。
- **Docker 容器化**：镜像分层缓存依赖；**模型权重**用大卷挂载或启动时从对象存储拉取，避免镜像过大。

与 [6.2 Docker 与 CI/CD](6.2_Docker_CICD.md) 配合做镜像构建与发布流水线。

# 4. 接口设计最佳实践

- **版本化路径**：如 `/v1/chat/completions`，兼容升级。
- **幂等与追踪**：对写操作使用 `Idempotency-Key`；全链路透传 `trace_id` / `X-Request-ID`。
- **错误模型**：统一 JSON 结构（code、message、details），勿泄漏堆栈到公网。
- **健康检查**：`GET /health` 区分 **存活**（进程起）与 **就绪**（模型已加载、依赖可达）。

# 5. 踩坑记录

- 在 `async` 路由里调用**阻塞**推理导致全局卡顿 → 改用线程池或独立推理服务。
- SSE 被 **Nginx 缓冲**吞掉 → 关闭 proxy_buffering 或改用合适缓存头。
- Pydantic 模型与 **可选字段默认值** 导致「未传参却生效」→ 区分 `None` 与缺省。
- 日志打印 **prompt 全文** 造成合规风险 → 脱敏或采样。
