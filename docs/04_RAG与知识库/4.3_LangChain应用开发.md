# 1. LangChain 是什么

**LangChain** 是一套面向 LLM 应用的**组件库与编排框架**（Python / JS），把模型调用、提示模板、检索、工具、记忆等拆成可组合模块。它不负责训练模型，而是解决「如何把 RAG、Agent、多步链路**工程化**」的问题。

- **适用场景**：快速搭建 PoC、统一不同模型厂商的接口、与向量库 / 文档加载器集成。
- **注意**：版本迭代快，生产环境应**锁定依赖**并抽象自己的领域层，避免业务与框架细节深度耦合。

# 2. 核心模块

## 2.1. Model I/O

- **ChatModel / LLM**：封装 OpenAI、Anthropic、本地 vLLM 等；统一 `invoke` / `stream` 接口。
- **PromptTemplate / ChatPromptTemplate**：变量注入与多消息角色；与线上 **chat template** 策略保持一致。
- **Output parsers**：将模型输出解析为 JSON、列表或 Pydantic 对象，减少手写正则。

## 2.2. Data Connection（文档→切块→向量化）

- **Document loaders**：PDF、HTML、Markdown、Notion 等；注意编码与元数据保留。
- **Text splitters**：按字符、token、标题层级切分；RAG 质量强依赖 **chunk 大小与重叠**。
- **Vector stores**：对接 Milvus、Chroma、FAISS 等；写入前确认 **嵌入维度与度量**。

## 2.3. Chains

将多个步骤串成 **LCEL（LangChain Expression Language）** 或经典 `Chain`：`检索 → 拼 prompt → 调模型 → 后处理`。适合固定流程的问答、摘要流水线。

## 2.4. Agents 与 Tools

- **Agent**：模型在循环中决定「调用哪个工具、传什么参数」；适合开放域任务。
- **Tools**：封装 HTTP、SQL、检索、计算器；需做好 **权限、超时与错误回传**，避免工具被滥用（与 [5.1 Agent 设计与工作流](../05_Agent开发/5.1_Agent设计与工作流.md) 呼应）。

## 2.5. Memory

- **ConversationBufferMemory** 等：多轮上下文管理；长对话需 **摘要记忆** 或 **外部会话存储**，避免窗口爆炸。

# 3. RAG 实战：构建问答链

下面为**概念示例**（具体 import 路径随 LangChain 大版本可能调整，以当前文档为准）。

```python
from langchain.chains import RetrievalQA
chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectorstore.as_retriever()
)
```

**落地要点**：

- `as_retriever(search_kwargs={"k": 5})` 控制召回条数；可换 **MMR** 减冗余。
- 自定义 `chain_type_kwargs` 传入 `prompt`，约束「仅依据上下文作答、禁止臆测」。
- 生产建议加 **引用片段 id** 与 **拒答策略**，便于审计（详见 [4.1 RAG](4.1_RAG检索增强生成.md)）。

# 4. Agent 工具调用（ReAct）

**ReAct** 类 Agent 交替进行「思考—行动—观察」。LangChain 提供多种 Agent 类型（ReAct、OpenAI tools、structured chat 等）。

- **系统提示**需明确工具描述与何时停止。
- **最大迭代次数**必设，防止死循环与费用失控。
- 与 **MCP** 生态结合时，可把 MCP 工具暴露为 LangChain Tool（参见 [5.2 MCP 协议](../05_Agent开发/5.2_MCP协议.md)）。

# 5. 踩坑记录

- **版本 0.x → 1.x** API 大变，旧教程代码直接复制常无法运行。
- **异步与同步混用**导致事件循环错误；高并发服务优先 `ainvoke` / `astream`。
- **Retriever 未过滤租户**，导致跨租户数据泄漏。
- **链路过长**难以单测；应拆子链并对检索与模型分别 mock。
