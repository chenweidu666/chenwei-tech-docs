# 1. 为什么需要对齐训练

预训练与纯 SFT 主要优化「似然」与示范行为，但**人类偏好**是多维的：有用性、诚实性、无害、简洁度等往往无法被单一 loss 完全表达。**对齐训练**引入偏好信号，使模型在多个回复中更倾向「更好」的那一种，从而缓解讨好、幻觉、不安全输出等问题。

- **RLHF**：用奖励模型 + 强化学习（如 PPO）直接优化生成策略。
- **DPO**：跳过显式奖励模型与 RL 环，用偏好对直接更新策略，实现更简单、训练更稳定（在不少设置下）。

二者常与 **SFT** 串联：SFT 学格式与基本能力，对齐阶段学「何为好答案」。

# 2. RLHF 流程

经典三阶段：**SFT 策略初始化 → 训练奖励模型（RM）→ 用 PPO 等算法微调策略**，并常用 **KL 散度**把新策略锚在 SFT 附近，避免崩坏。

## 2.1. 奖励模型训练

- **数据**：同一提示下的人类排序或打分（A 优于 B）；质量比数量更关键，需覆盖拒绝回答、多轮、工具场景等。
- **目标**：给定 (prompt, response)，输出标量奖励；或与 Bradley–Terry 等配对损失等价建模。
- **注意**：奖励黑客（reward hacking）——模型可能学会「骗过高分」而人类观感差；需迭代数据与评测。

## 2.2. PPO 强化学习

- **思路**：策略网络生成回复，用 RM 打分作为回报信号，通过 PPO 更新策略。
- **工程难点**：**方差大、训练不稳定**；需 careful 的 advantage 估计、batch 构成与 checkpoint 回滚。
- **算力**：往往要多卡并行 rollouts，与推理服务化类似需关注吞吐与显存。

## 2.3. KL 散度约束

- **目的**：限制新策略 \(\pi\) 相对参考策略 \(\pi_{\text{ref}}\)（常为 SFT）偏离过大，保留语言能力并减轻模式崩塌。
- **实现**：在目标函数中加 \(\beta \,\mathrm{KL}(\pi \,\|\, \pi_{\text{ref}})\)；\(\beta\) 过大则改进有限，过小则易不稳定。

# 3. DPO：更简单的对齐方法

**Direct Preference Optimization** 将偏好学习写成**分类式损失**，直接更新策略参数，无需在线采样与显式 RM（可从数据中隐式对应到最优奖励形式）。

## 3.1. DPO 原理（直觉）

对同一 prompt 下的 chosen / rejected 回复，模型被鼓励提高 chosen 的相对对数概率、压低 rejected。参考模型用于防止过度偏离，起到类似 KL 正则的效果。实现上常与 TRL 的 `DPOTrainer` 对齐。

## 3.2. 偏好数据格式

- 常见字段：`prompt`、`chosen`、`rejected`；多轮时需与模板一致拼成单条 transcript。
- **一致性**：标注指南要定义「何为好」（事实性优先还是简洁优先等），否则噪声会直接注入梯度。

## 3.3. 与 RLHF 对比

| 维度 | RLHF（PPO+RM） | DPO |
|------|----------------|-----|
| 流水线 | RM 训练 + 在线 RL | 偏好数据直接优化 |
| 稳定性 | 较敏感，需调参 | 通常更易训稳 |
| 灵活性 | 可接复杂奖励（含规则） | 偏好需可写成对 |
| 算力 | rollouts 成本高 | 通常低于 PPO 环 |

实践中可先 **DPO 快速迭代**；若需多源奖励或在线反馈，再考虑 RLHF 类方案。

# 4. 实践建议

- **数据先于算法**：偏好标注指南、去偏、困难负样本（hard negatives）比对算法细节更决定上限。
- **参考模型冻结**：DPO / RLHF 中 \(\pi_{\text{ref}}\) 一般固定为 SFT checkpoint，避免目标漂移。
- **评测**：除自动指标外，固定测试集上做**成对比较**与红队探针；对齐后复查 **下游任务** 是否受损。
- **与 SFT 数据混合**：少量高质量 SFT 或 KL 锚定可缓解「对齐后变笨」。
