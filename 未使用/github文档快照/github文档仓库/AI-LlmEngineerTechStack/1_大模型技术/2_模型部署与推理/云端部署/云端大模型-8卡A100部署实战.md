# 3. 实际部署案例：8 台 8 卡 A100 服务器部署方案

**技术栈**：
- **<span style="color: red; font-weight: bold">Triton Inference Server</span>**：模型服务化框架，负责接收请求、管理模型、调度推理
- **<span style="color: red; font-weight: bold">NVIDIA NIM</span>**：标准化的推理微服务
- **TensorRT-LLM Runtime**：推理运行时引擎
- **<span style="color: red; font-weight: bold">Docker</span>**：容器技术，将应用和依赖打包成镜像
- **<span style="color: red; font-weight: bold">Kubernetes (K8s)</span>**：容器编排平台，管理多个容器的部署、扩缩容、故障恢复

**核心知识点**：
1. **Docker 镜像构建**：
   - 基于 NVIDIA CUDA 基础镜像
   - 安装 TensorRT-LLM Runtime、<span style="color: red; font-weight: bold">Triton Inference Server</span>
   - 将模型文件（.engine）和配置文件打包进镜像

2. **Triton Model Repository**：
   - 模型存储目录：`model_name/version/model.engine`
   - 配置文件 `config.pbtxt`：定义输入输出格式、批处理策略、实例组（多 GPU 支持）
   - Triton 会自动发现并加载模型

3. **部署方式**：
   - **Docker**：单机部署，`docker run` 启动容器
     1. **选择基础镜像**：实际项目中我会直接选用官方的 `nvcr.io/nvidia/tritonserver:xx-trtllm-python-py3` 镜像作为基础，这个镜像里已经集成了 CUDA、TensorRT-LLM Runtime 和 Triton Inference Server，避免自己从纯 CUDA 镜像一步步装环境。  
     2. **准备模型仓库目录**：在本地整理好 `models/` 目录，例如 `models/qwen-7b/1/model.engine` 和对应的 `config.pbtxt`，这就是 Triton 的 Model Repository 结构。  
     3. **构建自定义镜像**：基于官方 Triton 镜像做一个很薄的定制层，把本地的 `models/` 目录拷贝进镜像（例如放到 `/models`），并在镜像里约定好启动时默认使用 `/models` 作为 `--model-repository`。  
     4. **在服务器上构建/拉取镜像**：将这个镜像在目标服务器上构建好，或者推到私有镜像仓库再在服务器上拉取，保证线上环境拿到的是"内置模型 + 内置配置"的推理服务镜像。  
     5. **使用 `docker run` 单机启动容器**：在有 GPU 的机器上通过 `docker run` 启动该镜像，关键是三点：挂载 GPU（例如使用所有 GPU）、映射好 HTTP/gRPC/metrics 端口（如 8000/8001/8002）、给容器起一个固定名字方便后续运维。  
     6. **验证服务是否可用**：容器跑起来后，通过 Triton 的健康检查接口和一次简单的推理请求，确认模型加载成功、接口可访问、延迟与显存占用在预期范围内，即完成单机 Docker 部署。
   - **Kubernetes**：集群部署，支持多实例、自动扩缩容、健康检查
      - **1. 统一基础镜像与模型仓库结构**  
      - 依然使用官方 `nvcr.io/nvidia/tritonserver:xx-trtllm-python-py3` 作为基础镜像，在其上做一层轻量定制：把本地整理好的 `models/` 目录（如 `models/qwen-7b/1/model.engine` + `config.pbtxt`）内置到镜像中的 `/models`。  
      - 镜像内约定启动命令使用 `/models` 作为 `--model-repository`，保证**无状态、可复制**，方便在集群里横向扩展多个副本。

      - **2. 推送镜像到镜像仓库**  
      - 将上述定制好的 Triton 镜像推送到公司私有镜像仓库（Harbor、阿里云镜像仓库等），K8s 集群中所有节点都从这个仓库拉镜像。  
      - 这样就可以**在不同节点上随时拉起同一个版本的推理服务实例**，与单机 Docker `docker run` 相对应的是 K8s 的 `Deployment/StatefulSet` 控制器。

      - **3. 编写 Deployment（或 StatefulSet）用于多实例部署**  
      - 使用 `Deployment` 定义推理服务的**副本数、副本模板、资源限制**，关键配置包括：  
         - `spec.replicas`：初始副本数（例如 2 或 3），支持后续 HPA 自动扩缩容。  
         - `resources.requests/limits`：配置 GPU/CPU/内存资源（结合 `nvidia.com/gpu` 资源类型，并在节点上安装 NVIDIA Device Plugin）。  
         - `nodeSelector` / `tolerations` / `affinity`：将服务调度到有 GPU 的节点上。  
         - `volumeMounts`：如果模型不做进镜像，也可以通过 PVC/NFS/OSS-Fuse 等挂载 `models/` 目录，再指给 `--model-repository`。  
      - 若模型加载时间非常长、需要稳定实例名与持久卷，也可以使用 `StatefulSet` 保证 Pod 有固定标识。

      - **4. 配置 Service 暴露 HTTP/gRPC/监控接口**  
      - 创建 `Service` 将 Triton 的 `8000/8001/8002` 端口映射出来：  
         - `8000`：HTTP inference  
         - `8001`：gRPC inference  
         - `8002`：metrics（Prometheus 监控）  
      - 内网调用场景下使用 `ClusterIP`；如需对集群外提供服务，可以再加 `Ingress` 或 `LoadBalancer`。  
      - 对上层业务而言，调用统一的 `Service` 域名即可，由 K8s 负责将流量分发到多个后端 Pod 实例上，实现**多副本负载均衡**。

      - **5. 健康检查（liveness/readiness/startup probes）**  
      - 在 `Deployment` 中为 Triton 容器配置健康检查，常见做法：  
         - **readinessProbe**：调用 Triton 的 `/v2/health/ready` 或 `/v2/models/{model}/ready`，只有当模型加载完成后才对外提供流量。  
         - **livenessProbe**：定期探测 `/v2/health/live`，如果进程异常或长时间无响应，由 K8s 自动重启容器。  
         - **startupProbe**（可选）：模型冷启动时间长时，先使用 `startupProbe` 放宽存活判定，避免在加载模型期间被误杀。  
      - 通过这些 Probe，确保**不健康实例自动剔除，恢复后自动接入**，提高集群整体可用性。

      - **6. 自动扩缩容（Horizontal Pod Autoscaler, HPA）**  
      - 基于 CPU/GPU 利用率或自定义指标（例如 QPS、请求排队长度、平均延迟）配置 HPA：  
         - 当请求量增大、负载增高时，HPA 自动增加 `Deployment` 的 `replicas`，在更多 GPU 节点上拉起 Triton Pod。  
         - 当流量下降时，自动缩容减少副本数，释放 GPU 资源。  
      - 如果需要更精细的 GPU 利用率控制，可以结合 Prometheus + 自定义 Metrics Adapter，以显存占用、推理吞吐等作为扩缩容信号。

      - **7. 滚动升级与灰度发布**  
      - 利用 `Deployment` 的滚动更新策略，配置 `maxUnavailable`/`maxSurge`，实现**无停机滚动升级**，在更新镜像或配置时按批次替换旧 Pod。  
      - 高风险模型版本或大改动时，可以通过分多个 `Deployment` + `Service`/`Ingress` 权重控制实现灰度发布（如 10% 流量新版本，90% 老版本），观察稳定后再全量切换。

      - **8. 端到端验证与观测性**  
      - 在集群中部署完成后，通过：  
         - 健康检查接口确认所有 Pod `READY`；  
         - 发起一次完整推理请求（HTTP/gRPC），验证多副本返回结果一致性；  
         - 通过 `/metrics` + Prometheus/Grafana 监控延迟、吞吐、错误率、GPU 利用率。  
      - 在压测验证扩缩容策略是否符合预期后，就可以认为完成了**支持多实例、自动扩缩容与健康检查的 Kubernetes 集群部署**。

## 3.1 硬件环境

**服务器配置（集群）**：
- **GPU**：8 台服务器 ×（8 × NVIDIA A100 80GB）= 共 64 张 A100（单机内通过 NVLink 高速互联）
- **CPU**：每台服务器 AMD EPYC 或 Intel Xeon（建议 64 核以上）
- **内存**：每台服务器 512GB+ DDR4
- **存储**：每台服务器 NVMe SSD 2TB+（用于模型与日志存储）
- **网络**：万兆/25G/40G 网卡，机内 NVLink + 机间高带宽网络，保证多机并行与服务访问带宽

## 3.3 完整部署方案

### 3.3.1 方案一：Docker 单机部署（推荐入门）

**适用场景**：单机部署，快速验证，适合中小规模生产环境

**部署架构**：
```
客户端请求
    ↓
Nginx (负载均衡，可选)
    ↓
Triton Inference Server (Docker 容器)
    ↓
TensorRT-LLM Runtime
    ↓
8 × A100 GPU (Tensor Parallelism)
```

**安装步骤**：

1. **安装基础环境**

```bash
# 1. 安装 NVIDIA 驱动和 CUDA
sudo apt update
sudo apt install -y nvidia-driver-525 cuda-toolkit-12-1

# 2. 安装 Docker
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

# 3. 安装 NVIDIA Container Toolkit
distribution=$(. /etc/os-release;echo $ID$VERSION_ID)
curl -s -L https://nvidia.github.io/nvidia-docker/gpgkey | sudo apt-key add -
curl -s -L https://nvidia.github.io/nvidia-docker/$distribution/nvidia-docker.list | \
  sudo tee /etc/apt/sources.list.d/nvidia-docker.list
sudo apt-get update
sudo apt-get install -y nvidia-container-toolkit
sudo systemctl restart docker

# 4. 验证 GPU 支持
docker run --rm --gpus all nvidia/cuda:12.1.0-base-ubuntu22.04 nvidia-smi
```

2. **安装 Triton Inference Server**

```bash
# 拉取 Triton 镜像（包含 TensorRT-LLM）
docker pull nvcr.io/nvidia/tritonserver:24.01-trtllm-python-py3

# 创建模型目录
mkdir -p /opt/triton/models
```

3. **准备模型文件**

```bash
# 假设模型已转换为 .engine 文件
# 目录结构：
# /opt/triton/models/
#   └── qwen-7b/
#       └── 1/
#           ├── model.engine
#           └── config.pbtxt
```

4. **配置 Triton Model Repository**

创建 `/opt/triton/models/qwen-7b/config.pbtxt`：

```protobuf
name: "qwen-7b"
platform: "tensorrt_llm"
max_batch_size: 128

input [
  {
    name: "text_input"
    data_type: TYPE_STRING
    dims: [ -1 ]
  }
]

output [
  {
    name: "text_output"
    data_type: TYPE_STRING
    dims: [ -1 ]
  }
]

instance_group [
  {
    count: 1
    kind: KIND_GPU
    gpus: [ 0, 1, 2, 3, 4, 5, 6, 7 ]  # 使用 8 张 GPU
  }
]

parameters {
  key: "tensor_parallelism"
  value: {
    string_value: "8"  # TP=8，8 张 GPU 并行
  }
}
```

5. **启动 Triton 服务**

```bash
docker run -d --gpus all \
  --name triton-server \
  --shm-size=1g \
  -p 8000:8000 \
  -p 8001:8001 \
  -p 8002:8002 \
  -v /opt/triton/models:/models \
  nvcr.io/nvidia/tritonserver:24.01-trtllm-python-py3 \
  tritonserver --model-repository=/models \
               --allow-gpu-metrics=true \
               --log-verbose=1
```

6. **验证服务**

```bash
# 检查服务状态
curl http://localhost:8000/v2/health/ready

# 查看模型列表
curl http://localhost:8000/v2/models
```

### 3.3.2 方案二：Kubernetes 集群部署（推荐生产环境）

**适用场景**：大规模生产环境，需要高可用、自动扩缩容

**部署架构**：
```
客户端请求
    ↓
Kubernetes Service (负载均衡)
    ↓
Triton Pod (多个实例)
    ↓
TensorRT-LLM Runtime
    ↓
8 × A100 GPU (通过 Device Plugin 分配)
```

**安装步骤**：

1. **安装 Kubernetes**

```bash
# 安装 kubeadm, kubelet, kubectl
sudo apt-get update
sudo apt-get install -y apt-transport-https ca-certificates curl
curl -fsSL https://packages.cloud.google.com/apt/doc/apt-key.gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-archive-keyring.gpg
echo "deb [signed-by=/etc/apt/keyrings/kubernetes-archive-keyring.gpg] \
  https://apt.kubernetes.io/ kubernetes-xenial main" | \
  sudo tee /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl
sudo apt-mark hold kubelet kubeadm kubectl
```

2. **安装 NVIDIA Device Plugin**

```bash
kubectl create -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/v0.14.0/nvidia-device-plugin.yml
```

3. **创建 Triton Deployment**

创建 `triton-deployment.yaml`：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: triton-server
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: triton-server
  template:
    metadata:
      labels:
        app: triton-server
    spec:
      containers:
      - name: triton
        image: nvcr.io/nvidia/tritonserver:24.01-trtllm-python-py3
        args:
          - tritonserver
          - --model-repository=/models
          - --allow-gpu-metrics=true
        resources:
          limits:
            nvidia.com/gpu: 8
          requests:
            nvidia.com/gpu: 8
        volumeMounts:
        - name: models
          mountPath: /models
        ports:
        - containerPort: 8000
        - containerPort: 8001
        - containerPort: 8002
      volumes:
      - name: models
        hostPath:
          path: /opt/triton/models
          type: Directory
---
apiVersion: v1
kind: Service
metadata:
  name: triton-service
spec:
  selector:
    app: triton-server
  ports:
  - port: 8000
    targetPort: 8000
    name: http
  - port: 8001
    targetPort: 8001
    name: grpc
  type: LoadBalancer
```

4. **部署服务**

```bash
kubectl apply -f triton-deployment.yaml
kubectl get pods -l app=triton-server
```

## 3.4 安装监控服务

1. **安装 Prometheus**

```bash
# 下载 Prometheus
wget https://github.com/prometheus/prometheus/releases/download/v2.45.0/prometheus-2.45.0.linux-amd64.tar.gz
tar xvfz prometheus-*.tar.gz
cd prometheus-*

# 配置 prometheus.yml
cat > prometheus.yml <<EOF
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'triton'
    static_configs:
      - targets: ['localhost:8002']  # Triton metrics endpoint
  - job_name: 'dcgm'
    static_configs:
      - targets: ['localhost:9400']  # DCGM exporter
EOF

# 启动 Prometheus
./prometheus --config.file=prometheus.yml
```

2. **安装 DCGM Exporter**

```bash
# 使用 Docker 运行 DCGM Exporter
docker run -d --gpus all \
  --name dcgm-exporter \
  -p 9400:9400 \
  nvcr.io/nvidia/k8s/dcgm-exporter:3.1.8-3.1.5-ubuntu22.04
```

3. **安装 Grafana**

```bash
docker run -d \
  --name grafana \
  -p 3000:3000 \
  -e "GF_SECURITY_ADMIN_PASSWORD=admin" \
  grafana/grafana:latest
```

访问 `http://localhost:3000`，用户名/密码：`admin/admin`

## 3.6 性能优化建议

1. **模型量化**：使用 INT8 量化，显存占用减少 50%
2. **批处理**：配置 `max_batch_size=128`，提高 GPU 利用率
3. **Tensor Parallelism**：8 卡 A100 建议 TP=8（每张 GPU 处理一层的一部分）
4. **KV Cache 优化**：启用 PagedAttention，支持更大的 batch size
5. **监控告警**：设置 QPS、延迟、GPU 利用率告警阈值
