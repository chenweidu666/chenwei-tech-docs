# 1. 背景与动机

### 为什么需要两只龙虾？

在单实例 OpenClaw 部署中，存在以下问题：

1. **风险集中** - 交易助手和日常助手混用，配置冲突风险高
2. **资源竞争** - 交易 heartbeat 和日常对话抢占同一 Gateway
3. **记忆污染** - 交易记录混入日常记忆，上下文混乱
4. **调试困难** - 问题定位复杂，日志混杂

**解决方案**：双实例隔离架构

### 同步脚本

**里斯本 sync-memory.sh**：
```bash

cd /root/.openclaw/workspace/GateClaw-Bot

# 配置 SSH
export GIT_SSH_COMMAND="ssh -i ~/.ssh/github_key -o IdentitiesOnly=yes"

# 拉取最新

cp -f memory/*.md /root/.openclaw/workspace/memory/

git push origin master

cd openclaw-dashboard
DASHBOARD_PORT=7860 DASHBOARD_ALLOW_HTTP=true node server.js &

# 5. 配置 Nginx 反向代理
apt-get install -y nginx
cat > /etc/nginx/sites-available/dashboard << 'EOF'
server {
    listen 80;
    location / {
        proxy_pass http://127.0.0.1:7860;
    }
}
EOF
ln -sf /etc/nginx/sites-available/dashboard /etc/nginx/sites-enabled/
systemctl restart nginx
```

## 经验总结

### 踩过的坑

1. **Git 冲突** - 两只龙虾同时写 MEMORY.md
   - ✅ 解决：明确写入方，另一方只读

2. **SSH 认证失败** - GitHub 推送失败
   - ✅ 解决：配置 `GIT_SSH_COMMAND` 环境变量

3. **Dashboard 强制 HTTPS** - 手机访问报错
   - ✅ 解决：`DASHBOARD_ALLOW_HTTP=true`

4. **Tailscale 中国区无 App** - 无法手机访问
   - ✅ 解决：Nginx 反向代理 HTTP

### 最佳实践

1. **职责清晰** - 每只龙虾只负责一个领域
2. **Git 中枢** - 所有同步通过 GitHub
3. **定期备份** - 每周导出记忆文件
4. **监控告警** - Gateway 宕机立即通知

### 未来优化

1. **实时通信** - WebSocket 桥接（替代 Git 异步）
2. **自动故障转移** - 一只挂了另一只接管
3. **负载均衡** - 对话请求自动分发
4. **统一 Dashboard** - 一个面板管理两只龙虾
