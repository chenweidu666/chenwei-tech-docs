# 1. 为什么需要大模型

## 1.1. 统一底座的优势

传统 NLP 常常是「一任务一模型」：每个场景单独建数据集、单独训练与维护一套参数，重复投入大。做法是先在超大规模通用语料上做 **自监督预训练**，得到相对通用的语言与知识底座，再按业务用 **SFT、RAG、工具调用、评测与数据闭环** 等拼成具体产品能力——**一套底座、多种落地形态**，减少反复从零训小模型、重复造轮子。

### 传统 NLP vs 大模型范式

| 维度 | 传统 NLP | 大模型范式 |
|------|---------|-----------|
| **模型数量** | 每任务一个模型 | 一个底座 + 多个适配层 |
| **数据准备** | 每任务单独标注 | 通用语料 + 少量任务数据 |
| **训练成本** | 重复训练，成本高 | 一次预训练 + 轻量微调 |
| **维护成本** | 多模型版本管理复杂 | 统一升级底座 |
| **能力迁移** | 跨任务迁移困难 | 天然具备迁移能力 |

```
传统 NLP 流程：
任务 A → 数据集 A → 模型 A → 部署 A
任务 B → 数据集 B → 模型 B → 部署 B
任务 C → 数据集 C → 模型 C → 部署 C

大模型范式：
          ┌→ 任务 A 适配层 → 部署 A
通用底座 ─┼→ 任务 B 适配层 → 部署 B
          └→ 任务 C 适配层 → 部署 C
```

## 1.2. 实际成本对比

**案例**：某电商公司需要部署 10 个 NLP 任务（情感分析、商品分类、客服问答等）

### 传统方案（10 个小模型）

```
训练成本：
- BERT-base × 10 = 10 × $500 = $5,000
- 数据标注：10 × $2,000 = $20,000
- 部署维护：10 × $300/月 = $3,000/月

总成本（首年）：$5,000 + $20,000 + ($3,000 × 12) = $61,000
```

### 大模型方案（1 个底座 + 适配层）

```
训练成本：
- 底座（复用开源）：$0
- 数据标注：10 × $500（少量样本）= $5,000
- LoRA 微调：10 × $100 = $1,000
- 部署维护：1 × $1,000/月 = $1,000/月

总成本（首年）：$5,000 + $1,000 + ($1,000 × 12) = $18,000

节省：$61,000 - $18,000 = $43,000（70%）
```

# 2. 规模带来的能力跃迁

## 2.1. 涌现能力（Emergent Abilities）

当参数量、数据量与训练算力超过一定阈值后，模型往往会表现出 **涌现能力**（如复杂推理、多步指令遵循、少样本迁移）：小规模时不稳定或缺失的行为，在规模化后变得可用。

### 典型涌现能力

| 能力 | 小规模模型 | 大规模模型 | 阈值 |
|------|-----------|-----------|------|
| **少样本学习** | 不稳定 | 稳定 | ~10B |
| **思维链推理** | 几乎无 | 强 | ~60B |
| **指令遵循** | 需微调 | 零样本 | ~100B |
| **代码生成** | 片段级 | 函数级 | ~30B |
| **多语言** | 主要英语 | 多语言均衡 | ~50B |

### 思维链（Chain-of-Thought）示例

**问题**：小明有 5 个苹果，他给了小红 2 个，又买了 3 个，现在有几个？

**小模型（直接回答）**：
```
答案：6 个
```

**大模型（带推理过程）**：
```
让我们一步步思考：
1. 小明初始有 5 个苹果
2. 给了小红 2 个：5 - 2 = 3 个
3. 又买了 3 个：3 + 3 = 6 个
答案：6 个
```

## 2.2. 涌现能力的工程意义

**何时选择大模型**：
- ✅ 需要复杂推理（数学、逻辑）
- ✅ 多步骤任务（规划、分解）
- ✅ 少样本场景（标注数据少）
- ✅ 开放域问答（知识覆盖面广）

**何时选择小模型**：
- ✅ 单一任务、数据充足
- ✅ 延迟敏感（<50ms）
- ✅ 成本敏感（高并发）
- ✅ 隐私要求（本地部署）

# 3. 缩放定律详解

## 3.1. Chinchua 缩放定律（2022）

DeepMind 的 Chinchua 论文提出了更精确的缩放定律：

$$L = \left(\frac{N_c}{N}\right)^{\alpha_N} + \left(\frac{D_c}{D}\right)^{\alpha_D}$$

其中：
- $L$ = 损失值（越低越好）
- $N$ = 参数量
- $D$ = 训练数据量（token 数）
- $N_c, D_c$ = 常数
- $\alpha_N, \alpha_D$ = 缩放指数

### 关键发现

**最优训练数据量**：
$$D_{opt} \approx 20 \times N$$

即：**1B 参数的模型应该用 20B token 训练**

| 模型规模 | 最优数据量 | 训练 FLOPs |
|---------|-----------|-----------|
| 1B | 20B tokens | 2e19 |
| 10B | 200B tokens | 2e21 |
| 70B | 1.4T tokens | 1.4e23 |
| 700B | 14T tokens | 1.4e25 |

## 3.2. 实际训练配置参考

### LLaMA 系列训练配置

| 模型 | 参数 | 训练数据 | 训练时长 | A100 小时 |
|------|------|---------|---------|----------|
| LLaMA-7B | 7B | 1T tokens | 21 天 | 184,320 |
| LLaMA-13B | 13B | 1T tokens | 21 天 | 368,640 |
| LLaMA-70B | 70B | 2T tokens | 48 天 | 1,720,320 |

**成本估算**（按 AWS A100 80GB $3.67/小时）：
- LLaMA-7B: 184,320 × $3.67 ≈ **$676,000**
- LLaMA-13B: 368,640 × $3.67 ≈ **$1,353,000**
- LLaMA-70B: 1,720,320 × $3.67 ≈ **$6,313,000**

# 4. 2026 年的高效缩放

## 4.1. 从「堆参数」到「高效缩放」

仅有「继续堆稠密参数」一条路径，已不足以描述近年的工程现实。**2025–2026 年前后**，讨论更多落在 **固定推理预算下的高效缩放**：

### 技术演进路线

```
2020-2022: 堆稠密参数
  └─ GPT-3 175B → 训练成本高、推理慢

2023-2024: 稀疏化 + 注意力优化
  ├─ MoE（Mixtral 8×7B）
  ├─ GQA（LLaMA-2-70B）
  └─ MLA（DeepSeek-V2）

2025-2026: 系统级优化
  ├─ 投机解码
  ├─ 权重量化（INT4/FP8）
  ├─ 算子融合
  └─ 数据与后训练配方
```

## 4.2. MoE（Mixture of Experts）

**核心思想**：用「总参数量大、每步只激活子集」换容量与专家分工。

### MoE 架构对比

| 模型 | 总参数 | 激活参数 | 专家数 | 每 token 激活专家 |
|------|--------|---------|--------|----------------|
| Mixtral 8×7B | 47B | 13B | 8 | 2 |
| Grok-1.5 | 314B | ~50B | 64 | 2 |
| DeepSeek-MoE | 16B | 2.4B | 64 | 4 |

**优势**：
- ✅ 容量大（总参数多）
- ✅ 推理快（只激活部分）
- ✅ 专业分工（不同专家处理不同知识）

**劣势**：
- ❌ 训练不稳定（需要特殊技巧）
- ❌ 显存占用高（需加载所有专家）
- ❌ 通信开销大（分布式场景）

## 4.3. GQA（Grouped-Query Attention）

**核心思想**：降低 K/V 头数，减少 KV Cache 占用。

### 注意力机制对比

| 机制 | Q 头数 | K 头数 | V 头数 | KV Cache |
|------|-------|-------|-------|----------|
| MHA | 64 | 64 | 64 | 100% |
| MQA | 64 | 1 | 1 | 3% |
| GQA | 64 | 8 | 8 | 25% |

**LLaMA-2-70B 实例**：
- MHA（如 LLaMA-1-65B）：KV Cache 48GB
- GQA（LLaMA-2-70B）：KV Cache 12GB
- **节省 75% 显存，推理速度提升 40%**

## 4.4. MLA（Multi-Token Latent Attention）

**DeepSeek-V2 创新**：进一步压缩 KV Cache。

### KV Cache 压缩对比

| 模型 | 上下文 | KV Cache | 压缩率 |
|------|--------|---------|--------|
| LLaMA-3-70B | 128K | 96GB | 1x |
| DeepSeek-V2 | 128K | 21GB | 4.6x |

**技术细节**：
- 将 K/V 投影到低维 latent space
- 解码时再投影回原维度
- 损失<1% 精度，节省 75%+ 显存

# 5. 工程上的取舍

## 5.1. 能力、时延、成本的三角平衡

```
         能力（Accuracy）
            / \
           /   \
          /     \
         /       \
        /         \
    成本 ───────── 时延
   (Cost)        (Latency)
```

**无法同时最优**：
- 高能力 + 低时延 → 成本高（需要更多 GPU）
- 高能力 + 低成本 → 时延高（需要更小批量）
- 低时延 + 低成本 → 能力低（需要更小模型）

## 5.2. 实际选型决策树

```
开始
 │
 ├─ 是否需要复杂推理？
 │   ├─ 是 → 选择 70B+ 模型（GPT-4、Claude、LLaMA-3-70B）
 │   └─ 否 → 继续
 │
 ├─ 延迟要求？
 │   ├─ <50ms → 选择 7B 模型 + 量化（INT4）
 │   ├─ 50-200ms → 选择 13B-30B 模型
 │   └─ >200ms → 选择 70B+ 模型
 │
 ├─ 预算限制？
 │   ├─ 低预算 → 开源模型（LLaMA、Qwen）+ 自建
 │   ├─ 中预算 → 云服务（DeepSeek、Moonshot）
 │   └─ 高预算 → 商业 API（GPT-4、Claude）
 │
 └─ 数据隐私？
     ├─ 必须本地 → 开源模型 + 本地部署
     └─ 可上云 → 商业 API
```

# 6. 实战案例

## 6.1. 案例 1：从 BERT 迁移到 LLaMA

**背景**：某金融公司原有 10 个 BERT 模型，维护成本高。

### 迁移前

```
模型列表：
- 情感分析 BERT-base
- 新闻分类 BERT-base
- 实体识别 BERT-large
- 问答匹配 RoBERTa
- ...（共 10 个）

问题：
- 每个模型单独部署，运维复杂
- 新任务需要重新标注数据训练
- 无法处理开放域问题
- 年维护成本：$100,000
```

### 迁移后

```
架构：
LLaMA-3-8B（底座）
  ├─ LoRA 情感分析
  ├─ LoRA 新闻分类
  ├─ LoRA 实体识别
  └─ 零样本问答

优势：
- 统一底座，升级简单
- 新任务只需少量样本
- 可处理开放域问题
- 年维护成本：$30,000（节省 70%）
```

## 6.2. 案例 2：选择 MoE 还是稠密？

**背景**：某 SaaS 公司需要部署代码生成助手。

### 方案对比

| 方案 | 模型 | 质量 | 延迟 | 成本/月 |
|------|------|------|------|---------|
| A | Mixtral 8×7B | 85 分 | 150ms | $800 |
| B | LLaMA-3-70B | 90 分 | 300ms | $2,000 |
| C | DeepSeek-V2 | 88 分 | 200ms | $1,200 |

### 最终选择：方案 C（DeepSeek-V2）

**理由**：
- ✅ 质量接近最优（88 分）
- ✅ 延迟可接受（200ms）
- ✅ 成本适中（$1,200/月）
- ✅ MLA 技术，显存占用低
