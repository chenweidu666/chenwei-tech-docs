# 1. 什么是 SFT

**监督微调（Supervised Fine-Tuning, SFT）** 指在已经过大规模预训练的语言模型上，用「指令—回答」或「多轮对话」等**成对标注数据**继续训练，使模型学会遵循格式、语气、领域知识与安全策略。与预训练「预测下一 token」不同，SFT 更强调**对齐任务形态**：把通用语言能力「折」成可上线的产品行为。

- **典型位置**：常作为 RLHF / DPO 的**前置阶段**（先学会基本对话与工具格式），也可单独用于垂直领域适配。
- **与 LoRA**：全参 SFT 成本高；参数高效微调见 [2.3 LoRA / QLoRA](2.3_LoRA_QLoRA高效微调.md)。

## 1.1. 何时优先做 SFT

- 需要**稳定输出格式**（JSON、函数调用、固定模板）。
- 领域术语、流程与预训练分布差异大，仅靠 RAG 仍不够「口吻一致」。
- 要在较小算力下快速验证数据配方与评测闭环。

## 1.2. 常见误区

- **数据量迷信**：低质、重复、互相矛盾的指令会拉低泛化；应先做去重与冲突消解。
- **忽视基座与词表**：基座语言分布与业务语言不一致时，先评估 tokenizer 切分与困惑度。
- **过长上下文堆砌**：SFT 阶段若序列过长，显存与收敛难度上升；可配合截断策略与课程学习。

# 2. 数据格式设计

指令数据需要与**训练框架**和**推理时 Chat Template**一致，否则会出现「训练时一种格式、部署时另一种格式」的分布偏移。

## 2.1. Alpaca 格式

常用于单轮指令，字段典型为 `instruction`、`input`（可选）、`output`。适合快速原型与开源数据集互转；多轮场景需自行扩展为列表结构或统一拼进 `instruction`。

## 2.2. ShareGPT 格式

以 `conversations` 数组描述多轮角色（如 `human` / `gpt`），贴近产品对话日志。做 SFT 时注意：**系统提示**是否单独字段、是否允许工具消息占位，与推理引擎（vLLM、TGI 等）的 message 角色名保持一致。

## 2.3. Chat Template

Hugging Face 等模型自带 `tokenizer.apply_chat_template`：训练与推理应使用**同一模板**（含特殊 token、BOS/EOS、工具调用槽位）。更换模板等价于改变输入分布，往往需要**重新微调或至少小规模适配**。

# 3. 训练配置

## 3.1. 学习率与调度策略

- **量级直觉**：全参 SFT 学习率通常**低于预训练**一个数量级量级（具体依基座与数据规模而定）。
- **调度**：线性 warmup + cosine decay 较常见；数据极少时可缩短训练步数并加强正则。
- **监控**：同时看 **train loss** 与 **held-out 评测**（以及生成样例），避免单纯过拟合训练集句式。

## 3.2. Batch Size 与梯度累积

显存不足时用**梯度累积**等价放大 batch；注意学习率与有效 batch 的缩放关系。长上下文场景下 per-device batch 往往只能为 1，需通过累积或多卡保证稳定梯度。

## 3.3. 混合精度训练

BF16（Ampere 及以上）或 FP16 可显著提速；注意 loss scaling 与梯度裁剪。QLoRA 等 4bit 底座需使用与之配套的优化器与稳定超参（见 2.3 篇）。

# 4. 训练框架选型

- **LLaMA-Factory**：配置驱动、支持多种模型与 PEFT，适合快速实验与中小规模 SFT。
- **Axolotl**：社区数据集与训练配方丰富，适合复现开源微调流程。
- **原生 Transformers `Trainer`**：可控性最高，便于嵌入自定义 loss、多任务与日志；工程上需自己补齐分布式与 checkpoint 策略。

选型时对照：**是否必须 DeepSpeed / FSDP**、**checkpoint 格式与推理引擎**、**多机稳定性**。

# 5. 评估方法

- **自动指标**：可选 ROUGE、BLEU（摘要类）、Exact Match（抽取类）；对开放对话仅作参考。
- **模型裁判（LLM-as-Judge）**：成本低、一致性需固定 rubric 与多 seed；重要场景应用**人工抽检**校准裁判偏差。
- **安全与拒答**：对越权、违法、隐私类探针单独建集，避免「讨好型」过拟合。
- **与线上一致**：评测时的 decoding（temperature、stop token）应尽量接近生产。

# 6. 踩坑记录

- **模板与数据角色不一致**导致推理乱码或复读。
- **系统提示泄漏到用户可见输出**（训练数据未清洗占位符）。
- **工具调用 JSON 不合法**：需在数据阶段约束可解析样本，或加结构化 loss / 校验重试流程。
- **灾难性遗忘**：全参 SFT 步数过大可能损伤通用能力；可用较小学习率、混合通用指令数据或采用 LoRA 限定更新子空间。
