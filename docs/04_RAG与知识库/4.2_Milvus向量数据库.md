# 1. 向量数据库的作用

在 [RAG 检索增强生成](4.1_RAG检索增强生成.md) 链路中，文档经嵌入模型变成**高维向量**，检索即 **ANN（近似最近邻）** 查询。向量数据库负责：**持久化、索引、过滤、扩缩容与运维**，使「亿级向量、毫秒级检索」在工程上可落地。

- **与关系库对比**：传统 B-tree 不适配高维相似度；向量库为 **cosine / L2 / IP** 等度量优化。
- **与内存 Faiss 对比**：Milvus 提供分布式、多副本、持久化与 SDK，更适合线上服务而非单次脚本。

# 2. Milvus 核心概念

- **Collection**：逻辑上的一张「向量表」，含多个 **Field**（主键、向量、标量元数据等）。
- **Partition**（可选）：按业务切分数据，便于冷热分层或权限隔离。
- **Index**：在向量字段上构建 ANN 结构；不同索引在 **召回率、构建时间、内存** 上权衡不同。

## 2.1. Collection / Field / Index

- **Field 类型**：`FLOAT_VECTOR`（主检索）、`VARCHAR`、`INT64` 等用于过滤（如 `doc_id`、`tenant_id`）。
- **Schema 设计**：主键建议稳定可回溯；元数据字段应覆盖过滤条件，避免「先搜向量再在应用层扫全表」。
- **一致性**：写入与检索的 **consistency level** 需与业务容忍延迟对齐（强一致更慢）。

## 2.2. 索引类型对比

| 索引 | 原理 | 适用场景 |
|------|------|---------|
| HNSW | 分层可导航小世界图 | 高精度、延迟敏感、内存充足 |
| IVF_FLAT | 倒排聚类 + 精确重算 | 大规模、可调 `nlist` / `nprobe` 平衡速度与召回 |
| IVF_SQ8 | 聚类 + 标量量化 | 省内存、可接受轻微精度损失 |
| IVF_PQ | 积量化 | 超大规模、强省内存，需调参 |

**实践**：先定 **度量方式**（与训练嵌入时一致，多为 cosine 或内积）；再按 QPS、延迟 SLO 与内存预算选索引；上线前用**真实查询集**扫 `nprobe`、`ef` 等参数。

# 3. Python SDK 实战

以下伪代码示意 **Milvus 2.x** 风格流程（以官方 `pymilvus` 为准，版本差异请查当前文档）。

## 3.1. 连接与创建集合

- 使用 `connections.connect` 指向 standalone 或集群地址。
- 定义 `CollectionSchema`、`FieldSchema`，指定向量维度与主键是否自增。
- 创建 collection 后执行 `load()` 再提供查询服务（部署模式不同步骤可能略有差异）。

## 3.2. 插入向量

- **批量插入**：控制 batch 大小，避免单次 RPC 过大；写入与索引构建可异步。
- **ID 映射**：保留 `id → 业务主键` 映射，便于回源文档与权限校验。

## 3.3. 构建索引

- 在向量列上 `create_index`，指定索引类型与参数（如 HNSW 的 `M`、`efConstruction`）。
- 大数据集可先灌数据再建索引，或按分区滚动构建，避免长时间阻塞。

## 3.4. 相似度查询

- `search` 时传入 `query_vectors`、`top_k`、可选 **expr** 标量过滤（如 `tenant_id == 1`）。
- 返回 `ids` 与 `distance` 后，在业务侧用主键拉原文；注意 **分数与度量** 的含义（越大越好还是越小越好）。

# 4. 在 RAG 中的完整链路

1. **文档切分** → chunk 与元数据（标题、章节、时间）。
2. **嵌入** → 与线上推理同一模型与归一化方式。
3. **写入 Milvus** → 带过滤字段，便于多租户与权限。
4. **检索** → `top_k` + 阈值过滤 + 可选 **reranker**（交叉编码器）。
5. **拼 prompt** → 将片段注入上下文，调用 LLM 生成。

与 [LangChain 应用开发](4.3_LangChain应用开发.md) 结合时，可用向量 store 适配器；核心仍是 **schema 与过滤策略** 要与产品一致。

# 5. 性能调优

- **参数扫描**：`nprobe`（IVF）、`ef`（HNSW）对延迟与召回影响极大，应用离线评测集网格搜索。
- **分段与副本**：读多写少场景水平扩展只读副本；写入高峰与索引构建错峰。
- **嵌入维度**：更高维不一定更好；在评测集上对比 **维度 vs 延迟**。
- **缓存**：热门 query 的向量或可缓存；注意租户隔离与失效策略。

# 6. 踩坑记录

- **度量与嵌入不一致**（cosine 模型却用 L2）导致排序错误。
- **未建索引或 collection 未 load** 导致查询极慢或失败。
- **过滤表达式**与字段类型不匹配；或过滤过宽导致扫描量大。
- **主键重复覆盖** silent overwrite；批量导入前做去重校验。
- **版本升级**：Milvus 1.x 与 2.x API 差异大，迁移需读官方升级指南。
