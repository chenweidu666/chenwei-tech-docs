# 1. Docker 在 AI 服务中的应用

容器把**依赖、CUDA 运行时、Python 版本**与业务代码打包成可复现单元，便于在多环境（开发机、CI、K8s）一致运行。AI 服务尤其依赖**驱动与 cuDNN 版本**与底层镜像对齐。

## 1.1. GPU 镜像选型

- 使用 **NVIDIA Container Toolkit**，基础镜像选用与驱动匹配的 **CUDA/cuDNN** 标签（如 `nvidia/cuda:12.x-runtime-ubuntu22.04` 再叠 Python）。
- 推理镜像可基于 **vLLM / PyTorch 官方**推荐组合，避免自行拼凑不兼容版本。

## 1.2. 多阶段构建

- **构建阶段**：安装编译依赖、下载 wheel、运行 `pip install`。
- **运行阶段**：仅复制虚拟环境或 site-packages 与入口脚本，**减小攻击面与镜像体积**。
- `.dockerignore` 排除 `__pycache__`、`.git`、大模型文件（改挂载）。

## 1.3. 模型文件挂载策略

- **不推荐**把数十 GB 权重打进镜像层；使用 **卷挂载**、**init 容器拉取** 或 **对象存储按需缓存**。
- 只读挂载避免容器内误改；多副本共享存储时注意 **锁与一致性**。

# 2. Bazel 构建系统

**Bazel** 适合**大规模单仓**、多语言（C++ / Python / Protobuf）与可复现构建；依赖图明确，**远程缓存**可显著加速 CI。

## 2.1. 为什么用 Bazel（典型背景）

- 单仓内 C++ 库、Python 服务、固件目标之间存在**统一构建与测试**需求。
- 希望 **增量构建**确定性高，避免「本地能编、CI 不能编」的环境漂移。

## 2.2. BUILD 文件基础

- `BUILD` 中声明 `cc_binary`、`py_binary`、`docker_image` 等目标（视规则集而定）。
- **可见性 `visibility`** 控制依赖边界；`WORKSPACE` / Bzlmod 管理外部仓库。

## 2.3. C++/Python 混合项目

- 用 `pybind_extension` 或统一 ABI 的扩展模块衔接；注意 **manylinux** 与 glibc 版本。
- Python 侧用 **虚拟环境或 hermetic Python** 与 Bazel 规则对齐，避免混用系统 pip。

# 3. CI/CD 流水线（GitHub Actions）

- **自动测试**：单元测试、静态检查（ruff、mypy）、对关键路径的集成测试。
- **镜像构建**：`docker buildx` 多架构（如需要）；镜像 tag 与 **git sha** 绑定便于回滚。
- **推送与部署**：推送到镜像仓库后，触发 K8s rollout 或 SSH 部署；配合 **蓝绿 / 金丝雀** 降低风险。

**密钥**：使用 OIDC 或仓库 Secrets，勿将凭证写入镜像层。

# 4. 工业混合语言项目实战（示例）

在 **C++/Python 混合、多目标产物** 的仓库中，典型收益包括：

- **统一门禁**：一次 PR 同时跑 C++ 单测与 Python 服务契约测试。
- **制品可追溯**：每个 release 对应可复现的 Bazel 目标与容器 digest。
- **联调缩短**：将「本地编不过 / 环境不一致」类问题前移到 CI，减少跨团队等待。

具体耗时与效率提升因团队与历史债务而异；落地时优先保证 **最小可运行流水线**（lint + test + build），再迭代缓存与并行。

# 5. 踩坑记录

- GPU 镜像在 **无 GPU 的 CI** 上构建失败 → 分离 CPU 构建阶段或使用 `docker build` 的 `--build-arg` 跳过 CUDA 检测。
- **层缓存失效**：频繁改 `requirements.txt` 导致全量重装 → 拆分依赖层与代码层。
- Bazel **沙箱与网络**：下载依赖需配置 `--repo_env` 或代理；CI 上开启远程缓存需鉴权。
- 大模型 **拉取超时** → 使用内网 registry 或分块同步，部署脚本重试与校验 checksum。
