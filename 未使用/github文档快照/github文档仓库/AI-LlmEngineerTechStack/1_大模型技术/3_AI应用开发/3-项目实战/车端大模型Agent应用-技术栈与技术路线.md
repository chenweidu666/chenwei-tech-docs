# 车端大模型Agent应用：技术栈与技术路线

> 对应简历核心项目：**车端大模型Agent应用 | 2025.06-2025.12**  
> 目标：基于小参数量模型设计车端可部署Agent架构，结合RAG实现知识问答，实现意图识别与工具调用并与车机系统对接，在车端算力与延迟约束下保障实时响应。

---

## 一、项目目标与能力边界

| 能力模块         | 目标描述 |
|------------------|----------|
| 知识问答         | 驾驶手册、政策法规等车载知识检索+生成，端到端响应≤2s |
| 意图识别与工具调用 | 查询导航、车辆状态等任务型指令识别，与车机API对接形成闭环 |
| 实时响应         | 首字延迟&lt;500ms，依赖车端量化与推理优化 |
| 部署形态         | 车端可部署（小参数量模型 + 量化），非纯云端 |

## 二、技术栈清单

### 2.1 模型与底座

| 类别           | 技术选型           | 说明 |
|----------------|--------------------|------|
| 基座模型       | Qwen-1.8B/7B、LLaMA-3B/7B、ChatGLM3-6B 等 | 小参数量、支持工具调用与多轮对话，便于车端/边缘部署 |
| 微调方式       | LoRA / QLoRA、SFT  | 用座舱对话、意图、工具调用数据做指令微调 |
| 推理框架       | 见下文「国产芯片车机：推理框架选型」 | 车端低精度、低内存推理 |
| 量化           | INT8 / INT4（GPTQ、AWQ、llm-compressor 等） | 降低显存/内存与延迟，保障实时响应 |

### 2.2 Agent 与 应用框架

| 类别           | 技术选型           | 说明 |
|----------------|--------------------|------|
| Agent 框架     | LangChain、LlamaIndex、自研轻量 Agent 编排 | 意图解析、工具调用编排、多轮对话与上下文管理 |
| 工具调用       | Function Calling / Tool Use（OpenAI 兼容或模型原生） | 导航、车辆状态、空调等车机 API 的封装与调用 |
| 意图识别       | 规则+槽位 / 小模型分类 / 大模型直接输出 | 区分“知识问答”与“工具调用”等意图并路由 |

### 2.3 RAG 知识问答

| 类别           | 技术选型           | 说明 |
|----------------|--------------------|------|
| 文档解析       | PyMuPDF、python-docx、PaddleOCR（扫描件） | 驾驶手册、政策法规等多格式文档解析 |
| 向量化         | BERT/BGE、M3E、text2vec 等 | 中文车载领域文本向量表示 |
| 向量库         | Faiss、Milvus Lite、Chroma、本地 SQLite+向量扩展 | 车端或边缘侧轻量检索 |
| 检索策略       | 稠密检索 Top-K、可选 Reranker | 召回手册/法规相关片段 |
| 生成           | 小参数量模型 + RAG 上下文 | 基于检索结果生成答案，控制幻觉与延迟 |

### 2.4 车机对接与工程

| 类别           | 技术选型           | 说明 |
|----------------|--------------------|------|
| 车机通信       | 车载总线/域控 SDK、gRPC、REST、MQTT、DDS 等 | 与导航、车身、座舱等域对接 |
| 运行时         | C++/Qt、Android 车载、QNX 等 | 与现有车机系统集成 |
| 部署形态       | 车载 SoC（如高通、地平线等）上本地推理，或混合云边 | 满足离线/弱网与延迟要求 |

### 2.5 国产芯片车机：推理框架选型

国产车规 SoC 多数为 **ARM CPU + 自研 NPU/BPU**，无 NVIDIA GPU，因此**不用 TensorRT**，而以 CPU 推理或厂商 NPU 推理为主。车机大模型 Agent 场景（小参数量、低延迟）推荐按芯片类型选择如下。

| 芯片类型 / 厂商 | 典型芯片 | 推荐推理框架 | 说明 |
|-----------------|----------|--------------|------|
| **地平线 (Horizon)** | Journey J2/J3/J5/J6 | **llama.cpp**（CPU）+ 可选 **OpenExplorer/BPU SDK**（若后续开放 LLM 算子） | 座舱常用 J5/J6，ARM 多核强；当前大模型多以 **llama.cpp（GGUF）** 在 CPU 上跑 1.8B～7B 量化模型，延迟与功耗可控；BPU 现多用于感知，LLM 支持视厂商工具链更新而定。 |
| **黑芝麻 (Black Sesame)** | A1000 系列 | **llama.cpp**（CPU）、**ONNX Runtime**（若厂商提供 NN 运行时） | ARM + NPU；LLM 端侧落地仍以 **llama.cpp** 或 **ONNX Runtime** 在 CPU 上为主，NPU 多用于视觉。 |
| **芯驰 (SemiDrive)** | X9 等座舱 SoC | **llama.cpp**（CPU）、**Android NNAPI**（若为 Android 车机） | 多核 ARM，部分方案为 Android 车机；**llama.cpp** 跨平台好集成；若系统为 Android，可评估 **NNAPI + 厂商 NPU 驱动**（需确认是否支持 LLM 类算子）。 |
| **瑞芯微 (Rockchip)** | RK3588 等 | **llama.cpp**（CPU）、**RKNN**（NPU，需确认 LLM 支持） | RK3588 等带 NPU，**RKNN** 多用于 CV；大模型建议先用 **llama.cpp** 在 CPU 上跑量化模型，再视 RKNN 对 Transformer 的支持情况考虑迁移。 |
| **华为 昇腾 (Ascend)** | 车载/边缘 Atlas 等 | **MindSpore Lite**、**CANN** | 若车机或边缘盒为华为生态，用 **MindSpore Lite** + **CANN** 做量化与推理；需将 PyTorch/ONNX 模型转换到昇腾。 |
| **寒武纪 (Cambricon)** | 车载 MLU | **Cambricon SDK**、**MagicMind** | 需按寒武纪工具链做模型转换与部署；LLM 支持情况需查厂商文档。 |
| **高通 (Qualcomm)** | 骁龙座舱 | **llama.cpp**、**SNPE**、**QNN** | 非国产，但车机常见；**llama.cpp** 通用；**SNPE/QNN** 可用 DSP/NPU 加速，需高通 SDK。 |

**车机大模型 Agent 场景下的通用建议：**

1. **优先用 CPU 推理 + 量化模型**  
   国产车规 SoC 的 ARM 多核性能足够跑 **Qwen-1.8B/7B、LLaMA-7B 等 INT4/INT8 量化**，**llama.cpp（GGUF）** 无厂商绑定、易集成、支持流式，首字延迟与端到端延迟都较易做到 &lt;500ms / ≤2s。

2. **需要 NPU 加速时**  
   先确认厂商是否提供 **LLM/Transformer 的 NPU 支持**（算子、示例、性能数据）；若暂无，仍以 **llama.cpp + CPU** 为主，NPU 留给感知等已有成熟方案的模块。

3. **与现有车机系统集成**  
   车机多为 **QNX、Android 或 Linux**，**llama.cpp** 提供 C/C++ API，便于与 C++/Qt 或 Android NDK 集成；若选型 **ONNX Runtime**，同样注意国产芯片上 ARM 与厂商 NN 库的适配。

4. **总结表（按框架）**  
   - **llama.cpp（GGUF）**：国产芯片车机**首选**，CPU 推理、小参数量 + 量化、无厂商锁、易对接车机。  
   - **ONNX Runtime**：通用、多芯片支持，国产芯片上以 ARM CPU 为主，部分厂商有 ONNX 或自定义 Runtime。  
   - **厂商 NPU SDK**（地平线 OpenExplorer、RKNN、昇腾 CANN、寒武纪 MagicMind 等）：用于感知/视觉优先；LLM 是否上 NPU 需看厂商能力与路线图。

### 2.6 性能与优化

| 类别           | 技术选型           | 说明 |
|----------------|--------------------|------|
| 量化与压缩     | INT8/INT4、KV Cache 量化、剪枝 | 降低内存与推理时延 |
| 推理优化       | 批处理、连续批处理、流式输出、首 token 优化 | 首字延迟&lt;500ms，端到端≤2s |
| 监控与指标     | 首字延迟、端到端延迟、意图准确率、工具调用成功率 | 用于量产达标与问题排查 |

## 三、技术路线（分阶段）

### 阶段一：底座与车端可部署推理

1. **基座选型**：在 Qwen/LLaMA/ChatGLM 等小参数量模型中选型，优先支持 Function Calling 与中文表现好的版本。
2. **车端推理链路**：使用 llama.cpp / TensorRT-LLM 等，在目标 SoC 上跑通 FP16/INT8 推理，测量显存/内存与首 token、逐 token 延迟。
3. **量化**：对选定模型做 INT8/INT4 量化（GPTQ/AWQ 等），在车端复测精度与延迟，确保首字延迟&lt;500ms 可达成。

### 阶段二：Agent 架构与意图识别、工具调用

1. **Agent 编排**：用 LangChain/LlamaIndex 或自研轻量框架，实现“用户输入 → 意图识别 → 路由（知识问答 / 工具调用）→ 执行 → 回复生成”的闭环。
2. **意图识别**：定义座舱意图集合（如：知识问答、导航、车辆状态、空调、音乐等），通过规则+槽位或小模型/大模型输出意图与槽位，并路由到 RAG 或工具调用。
3. **工具调用**：将导航、车辆状态、空调等车机能力封装为 Agent 可调用的工具（OpenAI 兼容或自定义 schema），在车机侧提供 API，Agent 根据意图与槽位调用并拼接结果生成回复。

### 阶段三：RAG 知识问答

1. **语料构建**：驾驶手册、政策法规、车型说明等文档做清洗、分块，按段落或语义块切分。
2. **向量化与检索**：用 BGE/M3E 等做向量化，写入 Faiss/Milvus Lite 等，部署在车端或边缘；查询时 Top-K 检索 + 可选 Reranker。
3. **生成与约束**：将检索到的片段作为上下文，用小参数量模型生成答案；通过 Prompt 与生成长度限制控制幻觉与响应时间，目标端到端≤2s。

### 阶段四：车机对接与闭环验证

1. **车机对接**：与导航、车身、座舱等域对接（总线/SDK/gRPC 等），实现工具调用的真实执行（如调起导航、读取车速/续航等）。
2. **联调与测试**：在台架或实车上做端到端测试，覆盖知识问答、导航、车辆状态等多轮对话与工具调用场景。
3. **指标与量产**：固化首字延迟、端到端延迟、意图准确率、工具调用成功率等指标，达到意图识别&gt;90%、知识问答≤2s，完成量产可行性验证。

## 四、技术路线简图（逻辑关系）

```
用户语音/文本
      ↓
 意图识别（规则/小模型/大模型）
      ↓
   ┌──┴──┐
   ↓     ↓
知识问答  工具调用（导航/车况/空调…）
   ↓     ↓
 RAG    车机 API
(检索+生成) (车机域对接)
   ↓     ↓
   └──┬──┘
      ↓
 回复生成与流式输出
      ↓
 首字<500ms / 端到端≤2s
```
